IMPORTANT CONSTRAINT
Do NOT implement or configure any Replit-specific backend infrastructure (no databases, no auth, no secrets manager, no web services). Replit is just the execution environment.

A) ORCHESTRATION: ONE MAIN ENTRYPOINT THAT ALWAYS PRODUCES review_queue.csv
If review_queue.csv is currently generated by a separate Python script (review_queue.py), do not rely on the user to run it manually.

Implement ONE of these patterns (choose what fits the repo cleanly):

OPTION 1 (preferred): Single orchestrated run
- Keep netflix2trakt.py focused on:
  - reading Netflix viewing history
  - matching + scoring
  - writing resolved.csv / needs_review.csv / skipped.csv
- From the same run, invoke the review queue generator code by importing review_queue and calling a function (do not shell out).
- review_queue.csv must be produced automatically on every run.

OPTION 2: Dedicated pipeline runner
- Create a new top-level entrypoint (e.g., run_pipeline.py) that:
  1) calls the netflix2trakt pipeline
  2) calls the review queue generator
  3) writes run_summary.txt and the log file
- The user runs exactly one command and always gets all outputs.

B) LOGGING (REQUIRED)
- Use Python’s standard logging module.
- Create one log file per run under a logs/ directory.
- Log levels:
  - INFO: run start/end, config values, input/output paths, row counts
  - WARNING: low-confidence matches, partial enrichment, retries
  - ERROR: per-row failures, parsing issues, enrichment failures, write errors
- Print only a short summary to stdout; the log file is the durable record.

C) EVERY INPUT ROW MUST BE ACCOUNTED FOR
- Assign each input row a stable original_row_id (line number or UUID) at ingest.
- Ensure every row ends in exactly one of:
  - resolved.csv
  - needs_review.csv
  - skipped.csv
  - failures.csv
- At run end, assert:
  - total_input_rows == resolved + needs_review + skipped + failures
- If not equal, log a loud ERROR and emit failures.csv rows for missing IDs.

D) RUN SUMMARY FILE (HUMAN-READABLE)
- Generate run_summary.txt for each run describing what happened.
- Place it alongside outputs.
